# @effect/ai-openai minimal-schema Responses client

## Request summary
Replace the current `@effect/ai-openai` client implementation that is backed by the generated Responses API client with a hand-written client built on minimal schemas, following the same direction as `@effect/ai-openai-compat` but targeting the OpenAI Responses API.

## Research summary

### Current repository state
- `packages/ai/openai/src/OpenAiClient.ts` currently constructs a `Generated.OpenAiClient` with `Generated.make(...)` and exposes it through `Service.client`.
- `OpenAiClient.createResponse`, `createResponseStream`, and `createEmbedding` all currently use generated request / response schemas and types.
- `OpenAiLanguageModel.ts` is already manually implementing request construction and response interpretation. Its dependency on generated request / response types is primarily type-level.
- `OpenAiEmbeddingModel.ts` also depends on generated embedding request / response types.
- `@effect/ai-openai-compat` already provides the target architecture pattern for the main client: a thin handwritten client built on `HttpClient` plus a minimal local schema surface.
- The current websocket mode inside `OpenAiClient.ts` depends on `client.client.httpClient.preprocess(...)`, so changing `OpenAiClient.client` from a generated client to a raw `HttpClient` requires an explicit websocket-mode update.
- Existing tests in `packages/ai/openai/test` assert behavior around request construction, error mapping, embeddings, and permissive SSE decoding of keepalive-like events.

### Repository constraints
- Follow existing repo patterns and the instructions in `.patterns/`.
- Do not manually edit autogenerated barrel files; use `pnpm codegen`.
- Localized package changes should run `cd packages/ai/openai && pnpm docgen`.
- Required validation for implementation: `pnpm lint-fix`, targeted tests, `pnpm check:tsgo`, and a changeset.

### Implementation discoveries (Task 1)
- Added a dedicated `OpenAiSchema.ResponseStreamEvent` unknown-event fallback via `Schema.declare`, constrained to `type: string` values that are **not** one of the known event literals. This preserves tolerant unknown-event decoding while ensuring malformed known events do not silently decode as unknown.
- Added focused schema tests in `OpenAiSchema.test.ts` that exercise direct JSON decoding plus `Sse.decodeDataSchema(OpenAiSchema.ResponseStreamEvent)` to confirm keepalive / future event tolerance in the stream decode path.

### Implementation discoveries (Task 2)
- Added `packages/ai/openai/src/OpenAiClientGenerated.ts` with a `Context.Service` whose service value is the generated `Generated.OpenAiClient` itself (not wrapped under a `client` field).
- `OpenAiClientGenerated.make` now mirrors the existing `OpenAiClient.make` generated-client wiring for base URL, bearer auth, organization / project headers, and `OpenAiConfig.transformClient` precedence (options transform first, context transform applied by generated client).
- Added focused tests in `OpenAiClient.test.ts` validating the generated-client path for auth headers, custom base URL handling, and `OpenAiConfig.withClientTransform` application after the options-level transform.

### Implementation discoveries (Task 3)
- Refactored `OpenAiClient.make` to use direct handwritten HTTP calls for `/responses` and `/embeddings`, decoding with `HttpClientResponse.schemaBodyJson(OpenAiSchema.*)` and preserving existing `Errors.mapHttpClientError` / `Errors.mapSchemaError` mappings.
- Introduced a shared `resolveHttpClient` step in the main client methods so `OpenAiConfig.transformClient` remains context-applied after the options-level transform, while `OpenAiClient.client` now exposes the configured `HttpClient.HttpClient` (base URL + auth/header wiring + options transform).
- Updated websocket mode request preprocessing to derive from the new `OpenAiClient.client` shape (no `.client.httpClient` access), while keeping the same websocket error-event mapping logic and tolerant unknown event decoding via `OpenAiSchema.ResponseStreamEvent`.
- Updated `OpenAiEmbeddingModel` to use `OpenAiSchema` embedding request/response types and to explicitly reject non-vector (string/base64) embeddings with `InvalidOutputError` to avoid accidental character-array coercion.
- For stream-consumer typing (`OpenAiLanguageModel`), the unknown-event fallback branch (`type: string`) required an explicit known-event type guard so exhaustive event handling remains type-safe while still preserving tolerant runtime behavior for future / keepalive events.

## Confirmed decisions
The following product decisions have now been confirmed:

1. `OpenAiClientGenerated` should expose the generated client directly as the service value.
2. Embedding request / response types should also move to `OpenAiSchema`.
3. `OpenAiClient.client` should expose the transformed `HttpClient`.
4. The embedding client types should accept both numeric vectors and string / base64 embeddings.

## Goals
- Replace the main `OpenAiClient` implementation with a handwritten client built on `HttpClient` and minimal local schemas.
- Add a public `OpenAiSchema` module that defines only the request / response / SSE / embedding shapes required by the handwritten client, `OpenAiLanguageModel`, and `OpenAiEmbeddingModel`.
- Change `OpenAiClient.Service["client"]` to expose the underlying transformed `HttpClient.HttpClient`.
- Preserve convenient access to the generated client through a new `OpenAiClientGenerated` module.
- Update `OpenAiLanguageModel` to use `OpenAiSchema` request / response types while continuing to use generated model-id types where needed.
- Preserve runtime behavior, request paths, transform precedence, error mapping, and tolerant streaming behavior.

## Non-goals
- Removing or replacing the full generated `Generated.ts` module.
- Rewriting the core runtime behavior of `OpenAiLanguageModel`.
- Reworking `OpenAiTool.ts` beyond import or type adjustments that may be needed to keep the package compiling.
- Expanding the minimal schema surface beyond the parts exercised by the main client, language model, embedding model, and websocket mode.

## Functional requirements

### 1. Add a new public `OpenAiSchema` module
Create `packages/ai/openai/src/OpenAiSchema.ts` with a minimal, explicit schema surface modeled after the handwritten approach used in `@effect/ai-openai-compat`.

#### Design requirements
- Define local `Schema` values and exported types for the request / response structures actually used by this package.
- Do not use generated request / response types as the public types of the main handwritten client.
- Keep the schema surface intentionally minimal and usage-driven.
- Export `OpenAiSchema` from the package barrel.

#### Required schema coverage
At minimum, `OpenAiSchema` must cover the shapes currently referenced by the main package code paths:

| Current generated concept | Planned minimal schema equivalent |
| --- | --- |
| `Generated.IncludeEnum` | `OpenAiSchema.IncludeEnum` |
| `Generated.Message.Encoded["status"]` | `OpenAiSchema.MessageStatus` |
| `Generated.InputContent` | `OpenAiSchema.InputContent` |
| `Generated.InputItem` | `OpenAiSchema.InputItem` |
| `Generated.Annotation` | `OpenAiSchema.Annotation` |
| `Generated.SummaryTextContent` | `OpenAiSchema.SummaryTextContent` |
| `Generated.ReasoningItem` | `OpenAiSchema.ReasoningItem` |
| `Generated.Tool` | `OpenAiSchema.Tool` |
| `Generated.CreateResponse["tool_choice"]` | `OpenAiSchema.ToolChoice` |
| `Generated.TextResponseFormatConfiguration` | `OpenAiSchema.TextResponseFormatConfiguration` |
| `Generated.CreateResponse` | `OpenAiSchema.CreateResponse` |
| `Generated.ResponseUsage` | `OpenAiSchema.ResponseUsage` |
| `Generated.Response` | `OpenAiSchema.Response` |
| `Generated.ResponseStreamEvent` | `OpenAiSchema.ResponseStreamEvent` |
| `Generated.CreateEmbeddingRequest` | `OpenAiSchema.CreateEmbeddingRequest` |
| `Generated.CreateEmbeddingResponse` | `OpenAiSchema.CreateEmbeddingResponse` |

#### Required request item / content variants
The minimal schema must cover the variants that are currently emitted or consumed by `OpenAiLanguageModel` and the handwritten client.

Required request-side content / item variants include at least:
- `input_text`
- `input_image`
- `input_file`
- `message`
- `reasoning`
- `item_reference`
- `function_call`
- `function_call_output`
- `local_shell_call`
- `local_shell_call_output`
- `shell_call`
- `shell_call_output`
- `apply_patch_call_output`
- `mcp_approval_response`

Required response-side output / metadata variants include at least:
- assistant `message` items with content parts such as `output_text`, `text`, `summary_text`, `reasoning_text`, `refusal`, and any input-style content parts that the current logic accepts
- tool/output item variants currently pattern-matched by `OpenAiLanguageModel`, including at least:
  - `apply_patch_call`
  - `code_interpreter_call`
  - `computer_call`
  - `file_search_call`
  - `function_call`
  - `image_generation_call`
  - `local_shell_call`
  - `mcp_call`
  - `mcp_list_tools`
  - `mcp_approval_request`
  - `shell_call`
  - `web_search_call`

#### Required stream event coverage
The minimal SSE event schema must explicitly cover the stream events that the current implementation consumes or tolerates, including at least:
- `response.created`
- `response.completed`
- `response.incomplete`
- `response.failed`
- `response.output_item.added`
- `response.output_item.done`
- `response.output_text.delta`
- `response.output_text.annotation.added`
- `response.reasoning_summary_part.added`
- `response.reasoning_summary_part.done`
- `response.reasoning_summary_text.delta`
- `response.function_call_arguments.delta`
- `response.function_call_arguments.done`
- `response.code_interpreter_call_code.delta`
- `response.code_interpreter_call_code.done`
- `response.apply_patch_call_operation_diff.delta`
- `response.apply_patch_call_operation_diff.done`
- `response.image_generation_call.partial_image`
- provider error events if the stream path expects them

#### Unknown-event fallback requirements
`OpenAiSchema.ResponseStreamEvent` must intentionally preserve the current tolerant streaming behavior.

Specifically:
- Known events must decode to typed variants.
- Unknown events must **not** become schema failures.
- Keepalive-style or future provider events must continue to flow through the stream unchanged.
- The fallback branch must preserve at least `type: string` plus additional payload fields.
- The fallback must be ordered or structured so that malformed known events do not silently decode as generic unknown events unless that is already the current behavior.

#### Embedding schema requirements
- `OpenAiSchema.CreateEmbeddingRequest` and `CreateEmbeddingResponse` must replace generated request / response types in the main client and embedding model.
- The schema should preserve the currently supported public request fields.
- The embedding response schema must accept both numeric vectors and string / base64 embedding payloads.
- The implementation must preserve current runtime behavior intentionally when one of those forms is returned, rather than accidentally changing semantics through incidental type widening.

### 2. Add `OpenAiClientGenerated`
Create `packages/ai/openai/src/OpenAiClientGenerated.ts` so users who still want the generated client can access it separately from the main handwritten client.

#### API shape
The new module should expose:
- a context service named `OpenAiClientGenerated`
- `make(options)`
- `layer(options)`
- `layerConfig(options?)`
- the same `Options` shape as the main client when feasible

#### Service semantics
- The service value should be the generated client itself, not a wrapper that nests it under another `client` property.
- The module must reuse the same auth, base URL, organization header, project header, and `OpenAiConfig.transformClient` behavior currently embedded in `OpenAiClient.make`.
- The module exists as a compatibility escape hatch; it does not change the new direction of the main `OpenAiClient` API.

#### Export requirements
- Keep exporting the existing `Generated` module.
- Add `OpenAiClientGenerated` as a public package export.
- Stop exposing the generated client from the main `OpenAiClient.Service["client"]` field.

### 3. Refactor the main `OpenAiClient` to use `HttpClient` + `OpenAiSchema`
Refactor `packages/ai/openai/src/OpenAiClient.ts` so the main service becomes a handwritten minimal-schema client.

#### Service interface requirements
Update the service interface so that:
- `client` is `HttpClient.HttpClient`
- `client` represents the **transformed HTTP client used internally** by the service methods
- `client` includes base URL prepending, auth headers, organization / project headers, and the same transform behavior the service methods use
- `client` is not the raw unmodified context client
- `createResponse` accepts `typeof OpenAiSchema.CreateResponse.Encoded`
- `createResponse` returns `readonly [body: typeof OpenAiSchema.Response.Type, response: HttpClientResponse.HttpClientResponse]`
- `createResponseStream` accepts the `OpenAiSchema.CreateResponse` request shape without the streaming flag and returns a stream of `OpenAiSchema.ResponseStreamEvent`
- `createEmbedding` accepts `typeof OpenAiSchema.CreateEmbeddingRequest.Encoded`
- `createEmbedding` returns `typeof OpenAiSchema.CreateEmbeddingResponse.Type`

#### Runtime invariants
The refactor must preserve these current behaviors:
- request path for responses remains `/responses`
- request path for embeddings remains `/embeddings`
- auth and header wiring remain unchanged
- `OpenAiConfig.transformClient` precedence remains unchanged
- error mapping continues to use `Errors.mapHttpClientError` and `Errors.mapSchemaError`
- stream handling remains tolerant of keepalive / unknown provider events
- method names and high-level API semantics remain stable apart from the intentional client-type change

#### Implementation requirements
Implementation should mirror the `openai-compat` handwritten pattern:
- construct a configured base `HttpClient`
- resolve the `OpenAiConfig` context override for transform behavior consistently with current semantics
- issue POST requests directly with `HttpClientRequest`
- decode JSON bodies with `HttpClientResponse.schemaBodyJson(...)`
- decode streaming events with `Sse.decodeDataSchema(OpenAiSchema.ResponseStreamEvent)` or an equivalent handwritten path if needed to preserve fallback behavior

### 4. Update websocket mode and internal stream typing
The websocket-mode code in `OpenAiClient.ts` is part of the migration scope.

Requirements:
- Replace generated request / response types in websocket mode with `OpenAiSchema` types where they refer to request or response payloads.
- Update the websocket request construction path so it no longer assumes `OpenAiClient.client` is a generated client with an internal `.httpClient` field.
- Preserve current websocket-mode behavior and service requirements.
- Preserve current error-event handling behavior in the websocket path.
- Update any exported event aliases such as `ResponseStreamEvent` to refer to `OpenAiSchema` types.

### 5. Update `OpenAiEmbeddingModel`
Refactor `packages/ai/openai/src/OpenAiEmbeddingModel.ts` so it depends on `OpenAiSchema.CreateEmbeddingRequest` and `OpenAiSchema.CreateEmbeddingResponse`, not generated request / response types.

Requirements:
- Config typing must move to `OpenAiSchema` request types.
- Provider-response typing must move to `OpenAiSchema` response types.
- Runtime behavior for request construction and result mapping should remain stable.
- Because the schema accepts both numeric and string / base64 embeddings, the implementation must intentionally preserve or explicitly handle both forms instead of accidentally treating string embeddings as character arrays.

### 6. Update `OpenAiLanguageModel` with a type-level migration to `OpenAiSchema`
Refactor `packages/ai/openai/src/OpenAiLanguageModel.ts` so request / response typing moves from generated request / response types to `OpenAiSchema`.

#### Scope requirements
- This should be type-level only unless a very small runtime adaptation is required to align with the new minimal schema names or shapes.
- Generated model-id types may continue to be used, including `Generated.ModelIdsResponses` and `Generated.ModelIdsShared`.
- Prompt metadata and response metadata augmentation types should move to `OpenAiSchema` equivalents for item status, annotations, and related fields.

#### Specific migration requirements
Update type references for:
- `CreateResponse`
- `InputItem`
- `InputContent`
- `Tool`
- `ToolChoice`
- `TextResponseFormatConfiguration`
- `Annotation`
- `ReasoningItem`
- `Response`
- `ResponseUsage`
- `ResponseStreamEvent`
- any status- or annotation-related references that currently point at generated request / response types

#### Compatibility requirement
The refactor should preserve the current request builder, response mapper, and streaming logic as much as possible. The primary intended outcome is to decouple those implementations from generated request / response types, not to change behavior.
